# Human-Readable Markdown Counterparts — Design Spec

**Date approved**: 2026-05-14
**Repo**: `github.com/host452b/trending-on-github`
**Status**: approved; ready for implementation planning

## 1. Goal

The dataset's per-snapshot JSON files (`data/{daily,weekly,monthly}/*.json`)
are good for machine consumption but not human-readable. Add three
cumulative Markdown files at the dataset root so a reader can skim
every snapshot chronologically without parsing JSON.

## 2. Non-goals

- Not replacing the JSON. JSON remains the authoritative dataset.
- Not replacing the `dashboard.ipynb`. The notebook is a single-page
  "current state" overview; the markdown files are the
  chronological archive.
- Not adding pagination, search, or HTML/CSS color gradients to the
  markdown. Plain pipe-tables only, for portability and `git diff`
  readability.
- Not building per-snapshot sidecar files. The aggregated approach is
  the chosen file structure.

## 3. Architecture

### 3.1 New files

```
trending-on-github/
├── build_markdown.py            # script: reads data/, writes the three .md files
├── tests/test_markdown.py       # unit tests
└── data/
    ├── daily.md                 # cumulative log of daily snapshots, newest first
    ├── weekly.md                # cumulative log of weekly snapshots, newest first
    └── monthly.md               # cumulative log of monthly snapshots, newest first
```

### 3.2 Modified files

- `.github/workflows/scrape.yml` — add a "Rebuild markdown" step after
  "Rebuild dashboard.ipynb"; extend `file_pattern` on the auto-commit
  to include `data/*.md`.
- `data/README.md` — add one paragraph pointing at the three new files.
- `README.md` (repo root) — list `build_markdown.py` alongside
  `build_dashboard.py` in the layout block.

## 4. File layout & cadence

| File              | Entry per … | Newest entry | Notes                                           |
|-------------------|-------------|--------------|-------------------------------------------------|
| `data/daily.md`   | each daily JSON (one per UTC day)        | top | One section per existing `daily/YYYY-MM-DD.json` |
| `data/weekly.md`  | each weekly JSON (one per ISO week)      | top | Section's table reflects the most-recent overwrite of that week's file |
| `data/monthly.md` | each monthly JSON (one per calendar month) | top | Same for the month bucket |

Each scrape regenerates all three from scratch by enumerating
`data/{daily,weekly,monthly}/*.json`. As the JSON dataset grows, the
markdown grows. Within-bucket overwrites (weekly/monthly) show up as a
single entry updated each day; intra-bucket drift is preserved in the
git history of the corresponding markdown file.

## 5. Markdown section format

```markdown
## 2026-05-14 — 🔥 Daily Trending (window `2026.05.14`)

Captured at `2026-05-14T00:30:00Z` · 11 repos · [raw JSON](daily/2026-05-14.json)

| # | Repo | Lang | ⭐ total | Forks | Period ⭐ | Description |
|--:|------|------|-------:|------:|---------:|-------------|
| 1 | [tinyhumansai/openhuman](https://github.com/tinyhumansai/openhuman) | Rust | 3,400 | 315 | **1,014** | Your Personal AI super intelligence. Private, Simple and extremely powerful. |
| 2 | … |

---
```

Header conventions per granularity:

| Granularity | Header line                                                                |
|-------------|----------------------------------------------------------------------------|
| daily       | `## YYYY-MM-DD — 🔥 Daily Trending (window `YYYY.MM.DD`)`                   |
| weekly      | `## YYYY-Www — 📅 Weekly Trending (window `YYYY.MM.DD-YYYY.MM.DD`)`         |
| monthly     | `## YYYY-MM — 🗓️ Monthly Trending (window `YYYY.MM.DD-YYYY.MM.DD`)`         |

The window string is taken verbatim from `period.label_compact` in the
JSON, so the markdown is internally consistent with the JSON it's
generated from. The first link in each section deep-links the raw JSON
for that snapshot.

## 6. Top-of-file structure

Each `.md` file starts with:

```markdown
# Daily trending — accumulated snapshots

Auto-generated by `build_markdown.py` from `data/daily/*.json` on
every scrape. **N snapshots** spanning `YYYY-MM-DD` → `YYYY-MM-DD`.
Raw JSON: [`data/daily/`](daily/).

---

<sections, newest first>
```

(Same template for weekly and monthly with appropriate emoji and
folder name.)

## 7. Escaping & robustness

- Pipe characters (`|`) inside repo descriptions are escaped to `\|`
  so they don't break the markdown table.
- Backslashes and backticks are passed through untouched (descriptions
  on GitHub very rarely contain them, and over-escaping makes the raw
  markdown harder to read).
- HTML in descriptions is *not* escaped at the markdown level — GitHub
  strips `<script>` and other dangerous tags when rendering README-style
  markdown. (Verified by the existing `dashboard.ipynb` which also
  re-emits trending HTML and renders safely.)
- If a JSON file is unreadable, the script logs a warning and skips
  it; the remaining files still produce a complete markdown.
- Atomic write: same pattern as `snapshot.py` — `tempfile.mkstemp` +
  `os.replace`, so a crashed run never leaves a half-written `.md`.

## 8. CI integration

Workflow diff:

```yaml
      - name: Rebuild dashboard.ipynb
        run: python build_dashboard.py
+     - name: Rebuild cumulative markdown
+       run: python build_markdown.py
      - uses: stefanzweifel/git-auto-commit-action@v5
        with:
          commit_message: "data: trending snapshot ${{ github.run_id }}"
          file_pattern: |
            data/**/*.json
            dashboard.ipynb
+           data/*.md
          commit_user_name: host452b
          commit_user_email: 32806348+host452b@users.noreply.github.com
```

The build script reads only the filesystem; no network. Adds roughly
0.5 s per cron run.

## 9. Testing

`tests/test_markdown.py` covers:

- **`escape_pipe`** — `"a|b"` → `"a\\|b"`, idempotent.
- **`build_section`** — given a snapshot dict + emoji + title, returns
  a string with the expected header, subtitle, table column count, and
  every row's full_name + period_stars.
- **`build_file`** — given a list of snapshots, returns a string with
  newest snapshot's date appearing before older snapshots' dates
  (chronological-newest-first).
- **`row_count`** — header summary correctly reports
  `N snapshots spanning X → Y`.
- **`empty_input`** — `build_file([])` produces a valid header with
  "0 snapshots" and no table sections; no exception.
- **Atomic write** — patching `os.replace` to raise verifies the
  previous `data/daily.md` is untouched and no temp file is leaked.

No network in tests. Fast (<1 s). No new pip deps.

## 10. Open questions

All resolved during brainstorming on 2026-05-14:

- ✅ File structure: aggregated per-granularity (3 files at `data/`).
- ✅ Within-bucket overwrites: one section per JSON file; daily
  rewrites the same entry until the bucket rolls. Git history
  preserves drift.
- ✅ Color gradients: not in markdown; plain pipe tables only. The
  notebook keeps the visual treatment.
- ✅ Build location: `build_markdown.py` at repo root, alongside
  `build_dashboard.py`.
